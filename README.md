# 如何更好地使用 AI 模型

更好地使用 AI，不只是写更好的 prompt，而是把工作环境改造成 AI 可以理解、调用、验证和复用的系统。

可以把它概括成四层：

```text
Context  决定 AI 理解什么
Tools    决定 AI 能做什么
CLI      决定 AI 能稳定重复什么
Verify   决定 AI 怎么知道做对了没有
```

也可以进一步表达为：

```text
.md / skills 负责知识
tools / API 负责能力
CLI 负责流程固化
tests / checks 负责反馈闭环
```

## 1. 上下文：让 AI 知道“为什么做、做成什么样”

上下文不只是资料，而是决策背景。AI 必须知道自己要做什么、为什么做、项目处在什么状态、哪些东西不能动，以及怎样才算完成。

好的上下文通常包括：

- 项目目标：这个项目最终要解决什么问题
- 当前状态：已经做了什么，哪些地方不能动
- 约束条件：技术栈、代码风格、权限边界、业务规则
- 成功标准：怎样算完成，怎样算失败
- 历史决策：为什么之前选择了某种方案

具体形式可以是：

- `README.md`
- `AGENTS.md`
- `CLAUDE.md` / `CODEX.md`
- `docs/architecture.md`
- `docs/workflows/*.md`
- skill 文件

关键是：上下文要可维护、可检索、可压缩。不要把所有信息都塞进一次 prompt，而是让 AI 能按需读取。

## 2. 工具：让 AI 有“手”可以执行任务

AI agent 只有语言能力是不够的。它必须有工具，才能真正完成任务。

工具可以分为几类：

- 文件工具：读写代码、搜索、修改、运行测试
- 浏览器工具：登录后的网页操作、截图、点击、抓取
- API 工具：GitHub、Linear、Slack、Gmail、数据库、云服务
- 本地 CLI：项目脚本、部署命令、数据处理命令
- 专用 MCP / plugin：把外部系统接进 agent

这里的关键不是“工具越多越好”，而是工具要满足这些条件：

- 权限清楚
- 输入输出稳定
- 错误信息可读
- 能被 AI 反复调用
- 尽量避免只能靠人眼判断的流程

例如，相比让 AI “去网页上手动点一遍”，更好的方式通常是给它 API 或 CLI。

## 3. CLI 化：把成功经验变成确定性能力

AI agent 的上下文是有限的，推理过程也有随机性。因此，重复性操作不应该长期依赖 prompt。

凡是已经跑通过多次的流程，都值得考虑 CLI 化：

- 创建项目
- 生成报告
- 同步数据
- 批量修改文件
- 发布版本
- 跑某类测试
- 检查某类规范
- 生成某类文档

好的 CLI 应该具备：

- 明确参数
- `--dry-run`
- 可重复执行
- 幂等性
- 结构化输出，比如 JSON
- 清楚的错误码
- 日志可追踪
- 有测试或 fixture

这样 AI 不需要每次重新理解完整流程，只需要调用稳定命令：

```bash
project-cli generate-report --date 2026-06-30 --format markdown
```

这会显著提高 AI 指令的准确度。

## 4. 反馈验证：让 AI 知道自己做得对不对

AI 最容易出问题的地方不是“不会做”，而是“做完以后不知道对不对”。

所以每个任务最好都有验证方式：

- 单元测试
- 类型检查
- lint
- snapshot
- 截图对比
- 数据校验
- schema 校验
- 人类验收标准
- CI 检查

理想状态是：

```bash
make test
make lint
make verify
```

然后 AI 根据验证结果继续修正。

## 总结

真正能把 AI 用好的人，核心不是写更玄的 prompt，而是搭建一个让 AI 可以理解上下文、调用工具、复用流程、验证结果的工程化环境。
